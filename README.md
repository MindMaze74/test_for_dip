# 🚀 Дипломный проект: Отказоустойчивая инфраструктура в Yandex Cloud

## 📋 Описание

Данный проект представляет собой автоматизированное развёртывание отказоустойчивой инфраструктуры для веб-сайта в облаке Yandex Cloud. Включает в себя:

- Веб-серверы (Nginx) с балансировкой нагрузки.
- Систему мониторинга на базе **Prometheus** и **Grafana**.
- Систему сбора и хранения логов на базе **Elasticsearch** и **Kibana**.
- Резервное копирование дисков (ежедневные снапшоты с хранением 7 дней).
- Единую точку входа через **Bastion** (SSH-шлюз и Nginx-прокси).

Вся инфраструктура описана как код с использованием **Terraform** и **Ansible**, что обеспечивает воспроизводимость и автоматизацию.

---
## Архитектура

- **Bastion** — единственная ВМ с публичным IP, выполняет роль:
  - SSH-шлюза
  - Прокси для сайта (порт 80)
  - Прокси для Grafana (порт 3000)
  - Прокси для Kibana (порт 5601)
- **Web-серверы** (2 шт) — в приватных подсетях, балансировка через Bastion
- **Prometheus** — сбор метрик
- **Grafana** — визуализация (admin/admin)
- **Elasticsearch** — хранение логов
- **Kibana** — просмотр логов
- **Snapshots** — ежедневное резервное копирование дисков (7 дней)

## Архитектура решения

### Сетевая структура

- Создан один **VPC** (`diplom-test-network`) с двумя зонами доступности (`ru-central1-a`, `ru-central1-b`).
- В каждой зоне имеются:
  - **Публичная подсеть** (`10.0.10.0/24` и `10.0.11.0/24`) – для **Bastion**.
  - **Приватная подсеть** (`10.0.1.0/24` и `10.0.2.0/24`) – для всех остальных сервисов.
- Для доступа в интернет из приватных подсетей настроен **NAT-шлюз**.
- **Security Groups** настроены по принципу «минимально необходимых прав»: все ВМ, кроме Bastion, принимают трафик только от Bastion.

### Состав виртуальных машин


| `bastion` | ru-central1-a | SSH-шлюз, Nginx-прокси (балансировщик для сайта, прокси для Grafana и Kibana)
| `web1`, `web2` | ru-central1-a, ru-central1-b | Веб-серверы (nginx в Docker) 
| `prometheus` | ru-central1-a | Сбор метрик (Prometheus в Docker)
| `grafana` | ru-central1-a | Визуализация (Grafana в Docker)
| `elasticsearch` | ru-central1-a | Хранилище логов (Elasticsearch в Docker)
| `kibana` | ru-central1-a | Просмотр логов (Kibana в Docker)

Все ВМ используют **прерываемые** инстансы для экономии бюджета.

### Сервисная модель

- **Веб-серверы**: Nginx в Docker-контейнерах, балансировка через Nginx на Bastion.
- **Мониторинг**: Node Exporter и Nginx Log Exporter на каждом веб-сервере; Prometheus собирает метрики; Grafana визуализирует их.
- **Логирование**: Filebeat (в Docker) на веб-серверах отправляет логи Nginx в Elasticsearch; Kibana предоставляет интерфейс для просмотра логов.
- **Резервное копирование**: ежедневные снапшоты дисков всех ВМ с хранением 7 дней (Terraform).

## Используемые технологии

| **Terraform** | Создание инфраструктуры (сети, ВМ, Security Groups, снапшоты). Аутентификация через сервисный аккаунт с постоянным ключом. |
| **Ansible** | Настройка ВМ: установка Docker, запуск контейнеров, настройка экспортеров, Filebeat, сервисов мониторинга и логирования. |
| **Docker** | Изоляция сервисов (nginx, node-exporter, nginx-log-exporter, filebeat, prometheus, grafana, elasticsearch, kibana). |
| **Yandex Cloud CLI** | Создание сервисного аккаунта и ключа, диагностика. |
| **Git** | Управление версиями кода (репозиторий на GitHub). |

## Принятые компромиссы и их обоснование

### 1. Ограничение на количество публичных IP-адресов
- **Проблема**: В Yandex Cloud действует квота на скорость создания публичных IP – не более 1 IP в минуту. Попытка создать несколько публичных IP одновременно приводила к ошибке `rate exceeded`.
- **Решение**: **Единственный публичный IP** выделен только для Bastion. Все остальные сервисы размещены в приватных подсетях. Доступ к ним организован через **Nginx-прокси на Bastion** (сайт, Grafana, Kibana).
- **Результат**: Квота не превышается, безопасность повышена, доступ к сервисам сохранён.

### 2. Отказ от Yandex Application Load Balancer
- **Проблема**: По ТЗ требуется балансировщик нагрузки для веб-серверов, но его создание требует дополнительного публичного IP.
- **Решение**: Балансировка реализована с помощью Nginx на Bastion. Это простое и надёжное решение, не нарушающее квоту.
- **Компромисс**: Балансировщик не является управляемым сервисом Yandex Cloud, но это допустимо для дипломного проекта и демонстрирует навыки администрирования.

### 3. Размещение Grafana и Kibana в приватных подсетях
- **Проблема**: По ТЗ они должны быть в публичных подсетях, но это потребовало бы дополнительных публичных IP.
- **Решение**: Они размещены в приватных подсетях, доступ организован через прокси на Bastion. Это не ухудшает функциональность и повышает безопасность.

### 4. Использование прерываемых (preemptible) ВМ
- **Решение**: Все ВМ созданы с параметром `preemptible = true`, что даёт экономию до 70% стоимости. Для дипломного проекта это допустимо, так как инфраструктура не является критичной к перезапускам.

### 5. SSH-доступ
- **Проблема**: На Bastion изначально не было приватного SSH-ключа для подключения к другим ВМ.
- **Решение**: Terraform передаёт публичный ключ на все ВМ через `ssh-keys`, а Ansible дополнительно распространяет ключи внутри сети. Таким образом, доступ автоматизирован.

## 🚀 Инструкция по развёртыванию

### Предварительные требования
- Установленный **Terraform** (>=1.0), **Ansible** (>=2.9), **Yandex Cloud CLI**.
- Настроенный профиль Yandex Cloud с правами на создание ресурсов.
- Сервисный аккаунт с авторизованным ключом (путь к ключу указан в `terraform.tfvars`).
- SSH-ключи на локальной машине (`~/.ssh/diplom` и `~/.ssh/diplom.pub`).
